# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Student team project for an "Architecture Logicielle" course: a news website (`site d'actualité`)
built in three parts that map to three team members (see `docs/plan-repartition.md`):

1. **Site Web (MVC)** — Servlets + JSP, the browser-facing UI.
2. **Services Web** — SOAP (JAX-WS) for user CRUD + auth, REST (JAX-RS) for articles (XML or JSON).
3. **Application Client** — a separate Java desktop app that authenticates over SOAP and manages users.

The authoritative spec is `docs/cahier-des-charges.pdf`; `README.md` and `docs/plan-repartition.md`
summarize it and assign files to members.

## Current state — read before editing

**The entire source tree is empty stubs (0-byte files).** The package/file layout exists and is
committed, but no implementation has been written yet, including `src/main/resources/schema.sql`,
`web.xml`, all `.jsp` views, `css/style.css`, and `client/pom.xml`. When asked to "implement X",

code consistent with the layering and naming rules below so the three parts integrate cleanly.

## Build & run

Two independent Maven projects:

- **Web app (root)** — `pom.xml`, packaging `war`, `finalName` `actualite` → produces `target/actualite.war`.
  ```bash
  mvn clean package        # build the .war
  ```
  Deploy the WAR to **Apache Tomcat 10+** (required — see Jakarta note below). There is no embedded
  server plugin configured.

- **Client (`client/`)** — a standalone Maven module. Its `pom.xml` is currently empty and must be
  authored before the client builds.
  ```bash
  cd client && mvn package
  ```

Java 17 (`maven.compiler.source/target = 17`). No tests, linter, or CI are configured yet; if you
add tests, JUnit must be added to `pom.xml` first.

## Architecture & conventions

- **Strict MVC / layered access.** A controller (Servlet) must never touch the database directly:
  `controller → service → dao → database`. Models are plain entities. This separation is graded, so
  preserve it. Same rule for SOAP/REST endpoints — they call `service`, not `dao`.

- **Jakarta EE 10, not Java EE.** All dependencies use the `jakarta.*` namespace (`jakarta.servlet`,
  `jakarta.ws.rs`, `jakarta.xml.ws`). Use `jakarta.*` imports everywhere — `javax.*` will not compile
  and forces Tomcat 9, which is wrong for this stack. This is why Tomcat 10+ is required.

- **French domain naming.** Package root is `sn.diop.actualite`; classes and DB columns use French
  (`Utilisateur`, `Categorie`, `Jeton`, `Role`, `AccueilServlet`, `GestionArticlesServlet`). Match
  this when adding code.

- **Token-based auth (`Jeton`).** SOAP user-CRUD is protected by a token generated by an admin.
  `util/JetonFilter` is the security filter that validates tokens; `dao/JetonDao` /
  `service/AuthService` back it. Three roles drive access: Visiteur (read-only), Éditeur (CRUD
  articles + categories), Administrateur (Éditeur + users + tokens).

- **REST content negotiation.** `rest/` services must be able to return both XML and JSON
  (`@Produces` with both media types); `rest/RestApplication` is the JAX-RS `Application` entry point.

- **Database.** MySQL via `mysql-connector-j`. JDBC connection is centralized in `dao/ConnexionBD`;
  the schema lives in `src/main/resources/schema.sql`.

## Layout

```
src/main/java/sn/diop/actualite/
  model/        entities (Article, Categorie, Utilisateur, Role, Jeton)
  dao/          data access (one DAO per entity + ConnexionBD)
  service/      business logic
  controller/   Servlets (public) + controller/admin/ (admin area)
  rest/         JAX-RS services
  soap/         JAX-WS services
  util/         JetonFilter, security helpers
src/main/webapp/WEB-INF/views/   JSP views (mirror controllers; admin/ subfolder)
src/main/resources/schema.sql    DB schema
client/                          separate Maven module: Swing client + SOAP client
```
